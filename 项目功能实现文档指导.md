# 基于微信小程序的可视化中医药查询系统功能实现文档指导

## 1. 文档说明

本文档依据《程家盛_202213407040_任务书》整理，用于指导“基于微信小程序的可视化中医药查询系统”的功能设计与实现落地。文档重点面向项目开发过程，强调功能拆分、页面规划、接口设计、数据结构、实施步骤与验收标准。

## 2. 项目目标

本项目拟构建一个基于微信小程序的中医药信息查询与可视化分析系统，为用户提供中药材、方剂、功效、适应症等内容的便捷检索与图形化展示能力。

系统总体目标如下：

- 通过微信小程序提供简洁直观的移动端使用入口。
- 建立中医药基础数据库，支持药材与方剂信息管理。
- 实现关键词查询、分类浏览等核心查询功能。
- 通过图表方式展示中医药数据特征，提高信息理解效率。
- 兼顾系统响应速度、可维护性与基本安全性。

## 3. 建设范围

结合任务书，项目功能范围建议划分为以下几部分：

- 微信小程序前端
- 后端服务接口
- 中医药业务数据库
- 数据可视化模块
- 数据管理与维护模块
- 拍照识别入口或扩展模块

其中，“拍照识别”在任务书模块设计中被列为主页面入口之一，如项目周期有限，可作为扩展功能分阶段实现，但页面入口与预留接口建议在一期中保留。

## 4. 系统总体架构

建议采用前后端分离架构：

- 前端：微信小程序，负责页面展示、交互逻辑、图表渲染与调用接口。
- 后端：提供 RESTful HTTP 接口，负责业务处理、查询服务、统计分析与数据校验。
- 数据库：存储药材、方剂、功效、适应症、分类、图表统计等基础数据。

推荐架构流程：

1. 用户在小程序中发起查询、浏览或可视化查看请求。
2. 小程序通过 HTTP 接口向后端发送参数。
3. 后端查询数据库并进行业务整理。
4. 后端将结构化 JSON 数据返回前端。
5. 前端将结果展示为列表、详情页或图表。

## 5. 用户角色定义

本项目可按简化角色设计：

### 5.1 普通用户

主要能力：

- 查询中药材信息
- 查询方剂信息
- 浏览分类内容
- 查看统计图表
- 使用拍照识别入口

### 5.2 管理员

主要能力：

- 维护药材数据
- 维护方剂数据
- 更新分类、功效、适应症等基础信息
- 管理统计口径与图表展示数据

如项目一期不开发独立管理后台，可通过数据库脚本、管理接口或本地导入方式完成数据维护。

## 6. 功能模块设计

### 6.1 首页模块

首页作为系统导航中心，建议包含以下内容：

- 系统标题与简介
- 搜索框快捷入口
- 中药查询入口
- 方剂查询入口
- 可视化分析入口
- 拍照识别入口
- 热门药材或常用方剂推荐区域

实现要点：

- 页面布局要简洁，突出高频功能。
- 首页应提供统一搜索入口，支持跳转到不同查询结果页面。
- 常用模块入口采用宫格或卡片式设计，便于移动端点击。

### 6.2 中药材查询模块

该模块是系统核心功能之一，主要支持中药材基础信息查询与展示。

### 主要功能

- 按药材名称搜索
- 按功效关键词搜索
- 按分类浏览
- 查看药材详情

### 药材详情建议字段

- 药材名称
- 别名
- 性味归经
- 功效
- 主治
- 用法用量
- 使用注意
- 来源说明
- 分类信息

### 页面设计建议

- 查询页：顶部搜索框 + 分类筛选 + 结果列表
- 详情页：基础信息卡片 + 功效说明 + 使用建议 + 关联图表

### 6.3 方剂查询模块

该模块用于展示方剂信息及组成分析。

### 主要功能

- 按方剂名称搜索
- 按功效或主治搜索
- 按分类浏览常见方剂
- 查看方剂详情与组成信息

### 方剂详情建议字段

- 方剂名称
- 组成药材
- 功效
- 主治
- 用法用量
- 适用说明
- 来源出处
- 方剂分类

### 页面设计建议

- 查询页支持关键词搜索与分类筛选
- 详情页可突出“方剂组成”与“功效主治”两个重点区域
- 可增加组成药材联动跳转，点击药材后进入药材详情页

### 6.4 数据可视化模块

该模块用于直观展示中医药数据规律，增强系统表达能力。

### 建议图表类型

- 柱状图：展示高频药材、常见功效数量分布
- 饼图：展示药材类别占比、方剂分类占比
- 折线图：如后续引入访问统计，可展示查询趋势
- 词云或排行图：展示高频关键词或常用药材

### 可视化主题建议

- 药材功效分布统计
- 常见方剂分类分布
- 方剂组成中高频药材分析
- 不同主治方向下的方剂数量对比

### 实现建议

- 前端采用 ECharts for 微信小程序。
- 后端优先返回已经聚合好的统计结果，减少前端计算压力。
- 图表需支持空数据提示、加载状态与错误状态。

### 6.5 拍照识别模块

任务书将拍照识别列为首页功能入口，建议将其设计为扩展模块。

### 一期建议

- 提供拍照上传入口
- 完成页面占位与交互流程
- 预留后端识别接口
- 支持返回识别结果后跳转到药材详情页

### 二期可增强内容

- 接入 OCR 或图像识别服务
- 对药材名称或标签文字进行识别
- 根据识别结果进行模糊匹配与推荐

若毕业设计周期有限，可在论文与答辩中说明本模块采取“预留接口 + 原型展示 + 后续扩展”的实现策略。

### 6.6 数据管理模块

数据管理模块主要服务于后端，负责中医药数据的统一维护与读取。

### 功能建议

- 药材数据新增、修改、删除、查询
- 方剂数据新增、修改、删除、查询
- 分类字典维护
- 统计分析所需基础数据维护
- 批量导入初始化数据

### 实现建议

- 采用结构化表设计，避免信息冗余。
- 药材、方剂、功效、分类之间建立明确关联关系。
- 为高频查询字段建立索引，如名称、分类、功效关键词。

## 7. 页面结构规划

建议小程序至少包含以下页面：

| 页面名称 | 页面用途 | 核心功能 |
| --- | --- | --- |
| 首页 | 系统导航与快捷入口 | 搜索、模块跳转、推荐内容 |
| 中药查询页 | 中药材列表查询 | 搜索、分类、列表展示 |
| 中药详情页 | 药材详细信息展示 | 功效、主治、用法、注意事项 |
| 方剂查询页 | 方剂列表查询 | 搜索、分类、列表展示 |
| 方剂详情页 | 方剂详细信息展示 | 组成、功效、主治、用量 |
| 可视化页 | 图表展示分析结果 | 柱状图、饼图、排行图 |
| 拍照识别页 | 拍照或上传识别 | 图片采集、识别结果展示 |

### 8.2 方剂表 `formula`

- `id`

## 9. 接口设计建议

后端接口应围绕查询、详情和统计三类能力展开。

### 9.1 药材查询接口

`GET /api/herbs`

建议参数：

- `keyword`：关键词
- `categoryId`：分类编号
- `pageNum`：页码
- `pageSize`：每页数量

返回内容：

- 列表数据
- 总数
- 当前页码

### 9.2 药材详情接口

`GET /api/herbs/{id}`

返回药材完整详情信息。

### 9.3 方剂查询接口

`GET /api/formulas`

建议参数与药材查询保持一致，便于前端统一封装。

### 9.4 方剂详情接口

`GET /api/formulas/{id}`

返回方剂详情与组成信息。

### 9.5 可视化统计接口

`GET /api/statistics/herb-efficacy`

返回药材功效分布数据。

`GET /api/statistics/formula-category`

返回方剂分类统计数据。

`GET /api/statistics/formula-herb-top`

返回高频药材排行数据。

### 9.6 拍照识别接口

`POST /api/recognition/herb`

建议返回：

- 识别结果名称
- 置信度
- 匹配到的药材 ID
- 推荐候选列表

## 10. 前端实现建议

### 10.1 技术选型

- 小程序原生框架
- WXML 构建页面结构
- WXSS 编写样式
- JavaScript 实现交互逻辑
- ECharts for 微信小程序实现数据可视化

### 10.2 前端目录建议

可按如下方式组织：

```text
miniprogram/
  pages/
    index/
    herb-list/
    herb-detail/
    formula-list/
    formula-detail/
    visualization/
    recognition/
  components/
    search-bar/
    herb-card/
    formula-card/
    chart-panel/
  services/
    api.js
  utils/
    request.js
    formatter.js
```

### 10.3 交互建议

- 查询输入支持防抖，减少频繁请求。
- 列表页支持分页加载或触底加载。
- 详情页展示应分区明确，避免大段文本堆叠。
- 图表区域需支持切换不同统计维度。

## 11. 后端实现建议

后端语言和框架可根据已有基础自由选型，重点保证接口清晰、结构稳定、便于演示。

建议满足以下要求：

- 提供统一接口前缀
- 统一返回格式
- 统一异常处理
- 参数合法性校验
- 支持跨端调试和日志记录

建议统一返回格式：

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

## 12. 性能与安全要求

依据任务书，系统需要兼顾性能与基本安全性，建议重点落实以下内容：

### 12.1 性能要求

- 高频查询结果可做本地缓存或短期接口缓存
- 统计图表数据优先后端聚合，减少前端运算
- 列表查询增加分页，避免一次返回大量数据
- 数据库建立必要索引，提高名称与分类检索效率

### 12.2 安全要求

- 对查询参数进行长度与格式校验
- 对上传图片进行类型和大小限制
- 后端避免直接拼接 SQL
- 对异常信息进行统一封装，避免泄露内部实现细节

## 13. 分阶段实现建议

为便于毕业设计推进，建议按以下阶段开发。

### 第一阶段：基础框架搭建

- 完成小程序项目初始化
- 搭建页面路由
- 建立后端基础工程
- 设计数据库表结构
- 准备初始中医药数据

### 第二阶段：核心查询功能实现

- 完成首页
- 完成中药查询与详情页
- 完成方剂查询与详情页
- 联调药材、方剂查询接口

### 第三阶段：可视化功能实现

- 接入图表组件
- 完成功效分布、分类统计、排行图实现
- 优化图表样式与移动端适配

### 第四阶段：扩展与优化

- 增加拍照识别入口及接口预留
- 完善缓存、异常提示与空状态页面
- 优化搜索体验与页面加载速度

### 第五阶段：测试与验收

- 完成功能测试
- 完成接口测试
- 完成页面兼容性和性能测试
- 准备论文截图、流程图与演示材料

## 14. 验收标准建议

项目完成后建议至少满足以下验收标准：

- 能在微信开发者工具中正常运行
- 首页可进入各个功能模块
- 中药材信息可按名称或分类查询
- 方剂信息可按名称或分类查询
- 详情页能展示完整核心字段
- 可视化页面至少展示 2 到 3 类有效图表
- 系统交互流程完整，无明显报错
- 响应速度满足演示需要

## 15. 开发风险与应对建议

### 风险一：中医药数据来源不统一

应对建议：

- 优先选择结构较清晰的数据源
- 手动整理一套可演示的基础样本数据
- 在论文中说明数据整理方法与范围

### 风险二：图表数据不足导致展示效果弱

应对建议：

- 优先准备能够体现分类与分布特征的数据
- 先做静态统计口径，再逐步扩展动态分析

### 风险三：拍照识别难度较高

应对建议：

- 一期先完成入口、交互流程与接口预留
- 二期再接入第三方 OCR 或识别服务

## 16. 最终交付建议

为保证项目完整度，建议最终形成以下成果：

- 微信小程序前端工程
- 后端服务工程
- 数据库设计文档或 SQL 脚本
- 项目功能实现文档
- 系统测试说明
- 论文配套截图和演示材料

## 17. 结语

本项目的重点不在于构建复杂的医学推理系统，而在于完成一个结构清晰、功能完整、体验良好、具有可视化特色的中医药查询平台。因此开发时应优先保证“查询可用、展示清晰、图表直观、流程完整”，在此基础上再逐步增强拍照识别、统计分析与后台管理能力。
