# 拍照识别功能第一阶段开发任务清单

## 1. 第一阶段目标

第一阶段仅实现“单味中药饮片/药材外观图片识别”的基础闭环，目标如下：

1. 用户可以在小程序中拍照或选择图片
2. 前端可以将图片上传到后端识别接口
3. 后端可以调用视觉模型返回候选识别结果
4. 后端可以将识别结果匹配到本地药材库
5. 前端可以展示识别结果与候选结果
6. 匹配成功时可以跳转到已有药材详情页
7. 识别失败时可以提示用户重新拍摄或重新上传

---

## 2. 第一阶段识别范围

### 2.1 识别对象

第一阶段只支持以下场景：

- 单张图片
- 图片中以单味药材为主
- 识别对象为实物饮片或药材外观

第一阶段不处理以下情况：

- 多味药材同图
- 包装遮挡严重
- 强反光、严重模糊、主体过小
- 非药材图片

### 2.2 第一阶段药材白名单

第一阶段建议固定为以下 9 种药材：

| 标准名 | 别名 |
| --- | --- |
| 黄芪 | 北芪 |
| 当归 | 秦归 |
| 甘草 | 国老 |
| 桂枝 | 嫩桂枝 |
| 白芍 | 杭芍 |
| 茯苓 | 云苓 |
| 柴胡 | 北柴胡 |
| 人参 | 园参 |
| 生姜 | 鲜姜 |

### 2.3 第一阶段识别规则

- 视觉模型只允许在白名单范围内输出结果
- 后端返回 Top-1 结果和 Top-3 候选结果
- 如果无法判断，则返回“未识别”
- 如果模型结果无法匹配数据库，则返回未匹配状态

---

## 3. 第一阶段业务流程

```text
用户选择图片或拍照
    ->
前端预览图片
    ->
用户点击开始识别
    ->
前端上传图片到后端
    ->
后端调用视觉模型
    ->
后端匹配 herb 表
    ->
前端展示识别结果
    ->
用户点击查看药材详情
```

---

## 4. 数据库任务

## 4.1 现有表复用

第一阶段直接复用现有药材表：

- `herb`

匹配字段：

- `name`
- `alias`

第一阶段不要求修改 `herb` 表结构。

## 4.2 第一阶段是否必须新增表

第一阶段核心识别闭环**不强制要求新增数据库表**。

如果第一阶段需要保留识别记录，则新增一张识别记录表：

- `recognition_record`

## 4.3 `recognition_record` 表设计

| 字段名 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| id | bigint | 是 | 主键 |
| image_path | varchar(255) | 否 | 图片保存路径 |
| image_sha256 | varchar(64) | 否 | 图片摘要 |
| source_type | varchar(20) | 是 | 图片来源：camera / album |
| status | varchar(20) | 是 | success / no_match / failed |
| recognized_name | varchar(100) | 否 | 模型返回的首选药材名 |
| matched_herb_id | int | 否 | 匹配到的药材 ID |
| confidence | decimal(5,4) | 否 | 首选结果置信度 |
| candidate_json | json 或 text | 否 | 候选结果列表 |
| error_message | varchar(255) | 否 | 失败信息 |
| created_at | datetime | 是 | 创建时间 |
| updated_at | datetime | 是 | 更新时间 |

建议索引：

- `idx_recognition_record_status`
- `idx_recognition_record_matched_herb_id`
- `idx_recognition_record_created_at`

## 5. 后端开发任务

### 5.1 新增依赖

修改文件：

- `backend/requirements.txt`

新增依赖：

- `python-multipart`
- `httpx`

### 5.2 新增识别路由

新增文件：

- `backend/app/api/recognitions.py`

修改文件：

- `backend/app/main.py`

新增接口：

- `POST /api/recognitions/herb-image`

请求类型：

- `multipart/form-data`

请求参数：

| 参数名 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| file | file | 是 | 图片文件 |
| sourceType | string | 否 | `camera` 或 `album` |

### 5.3 后端处理流程

接口内部按以下顺序实现：

1. 校验图片格式
2. 校验图片大小
3. 读取图片内容
4. 调用视觉模型
5. 获取 Top-1 和 Top-3 候选
6. 标准化药材名称
7. 匹配本地 `herb` 表
8. 组装返回数据
9. 如果启用识别记录，则写入 `recognition_record`

### 5.4 图片校验规则

- 支持格式：`jpg`、`jpeg`、`png`
- 文件大小上限：5MB
- 空文件直接拒绝
- 非图片文件直接拒绝

### 5.5 模型服务层

建议新增文件：

- `backend/app/services/model_client.py`
- `backend/app/services/recognition_service.py`

职责划分：

- `model_client.py`
  - 调用视觉模型接口
- `recognition_service.py`
  - 执行识别业务流程
  - 处理名称标准化
  - 处理数据库匹配
  - 组装返回结构

### 5.6 模型输出要求

模型输出必须可解析为固定 JSON，至少包含：

```json
{
  "top1_name": "黄芪",
  "top1_confidence": 0.87,
  "top1_reason": "颜色和纹理与黄芪较接近",
  "candidates": [
    {
      "name": "黄芪",
      "confidence": 0.87,
      "reason": "颜色和纹理符合"
    },
    {
      "name": "甘草",
      "confidence": 0.42,
      "reason": "根片外观有相似性"
    },
    {
      "name": "白术",
      "confidence": 0.18,
      "reason": "片形部分接近"
    }
  ]
}
```

### 5.7 名称匹配规则

匹配逻辑按以下顺序实现：

1. 按 `herb.name` 精确匹配
2. 按 `herb.alias` 匹配
3. 如果无法匹配，返回未匹配结果

建议封装函数：

- `normalize_herb_name(name: str) -> str`
- `match_herb_by_name_or_alias(db, name: str)`

### 5.8 接口返回结构

建议返回格式：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "recordId": 1,
    "matched": {
      "herbId": 1,
      "name": "黄芪",
      "alias": "北芪",
      "confidence": 0.87,
      "reason": "颜色和纹理与黄芪较接近"
    },
    "candidates": [
      {
        "name": "黄芪",
        "confidence": 0.87,
        "reason": "颜色和纹理符合"
      },
      {
        "name": "甘草",
        "confidence": 0.42,
        "reason": "根片外观有相似性"
      },
      {
        "name": "白术",
        "confidence": 0.18,
        "reason": "片形部分接近"
      }
    ],
    "unmatchedName": null
  }
}
```

### 5.9 后端异常处理

第一阶段至少处理以下异常：

- 图片格式不支持
- 图片过大
- 图片读取失败
- 模型接口超时
- 模型返回结构异常
- 数据库未匹配

## 6. 前端开发任务

### 6.1 修改页面

修改文件：

- `pages/recognition/index.js`
- `pages/recognition/index.wxml`
- `pages/recognition/index.wxss`

### 6.2 页面状态字段

建议增加以下状态：

- `imageUrl`
- `uploading`
- `analyzing`
- `result`
- `candidates`
- `errorText`

### 6.3 页面交互任务

第一阶段页面需要完成以下交互：

1. 选择图片
2. 预览图片
3. 点击“开始识别”
4. 识别中显示加载状态
5. 识别成功后展示首选结果
6. 展示候选结果列表
7. 支持点击“查看详情”
8. 支持点击“重新选择图片”

### 6.4 页面展示内容

识别结果区至少展示：

- 首选药材名称
- 识别置信度
- 简短识别依据
- 候选结果列表
- 查看详情按钮

### 6.5 前端上传接口封装

修改文件：

- `services/api.js`

新增方法：

- `recognizeHerbImage(filePath, sourceType)`

实现要求：

- 使用 `wx.uploadFile`
- 请求地址为 `/api/recognitions/herb-image`
- 支持上传 `file`
- 支持上传 `sourceType`

### 6.6 前端结果处理规则

- `matched.herbId` 存在时，显示“查看详情”
- `matched` 为空但有 `candidates` 时，展示候选结果
- 未识别时，提示重新拍摄
- 网络失败时，提示重试

### 6.7 跳转逻辑

匹配成功后跳转到：

- `/pages/herb-detail/index?id={herbId}`
